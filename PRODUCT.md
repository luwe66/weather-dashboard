## 一、项目起源

在 Kiro 中与 AI 对话，决定做一个天气数据可视化大屏项目。

---

## 二、核心决策

### 数据来源

调研了以下免费天气 API：

| API | 免费额度 | 备注 |
|-----|---------|------|
| Open-Meteo | 完全免费，无需注册 | 无需 API Key |
| OpenWeatherMap | 1000次/天 | 需注册 |
| WeatherAPI | 100万次/月 | 需注册 |
| 和风天气 | 1000次/天 | 中文支持好，数据覆盖全国 |
| 彩云天气 | 有免费层 | 分钟级降水预报准确 |

**决策：选用和风天气**
- 中文支持好
- 数据覆盖国内城市完整
- 免费额度（1000次/天）对大屏场景足够
- 文档完善，开发体验好

---

### 技术栈

对比了三种方案：

- **原生 HTML/JS**：门槛低，但大项目难维护
- **React**：生态大，全球最流行，稍复杂
- **Vue**：国内生态成熟，语法直观

**决策：Vue 3 + ECharts**
- 国内大屏项目 Vue 生态更成熟
- ECharts 是百度出品的图表库，中文文档完善，图表效果好
- 大屏 + ECharts 是非常成熟的技术组合
- 项目本身就需要用到 ECharts 图表

### API 用量控制模块

背景：和风天气按量计费，每月前 5 万次免费，超出后收费。

- [ ] **请求开关**：全局开启/关闭自动刷新，关闭后停止所有 API 请求，避免意外消耗额度
- [ ] **自动关闭保护**：当月请求累计达到 30,000 次时，自动关闭请求开关并提示用户
- [ ] **历史用量统计**：记录每个月的请求次数，用 ECharts 柱状图展示各月用量趋势，不清零、长期保留
- [ ] **当月用量进度**：显示本月已用请求数 / 5万，直观感知距离免费额度的距离

实现思路：
- 数据结构：按月存储，格式如 `{ "2026-05": 128, "2026-06": 340, ... }`
- 持久化到 localStorage，长期累计，不自动清零
- 开关状态同样持久化，刷新页面后保持上次设置
- 用量图表可复用项目已有的 ECharts，风格统一
